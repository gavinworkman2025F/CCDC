First 10 Minutes — Flag Drop Checklist

Red team warning: Backdoors, SSH keys, new accounts, and keyloggers are added in minutes 0–5. You have first access — use it or lose it. If you're not faster than this, you're playing defense from behind for 2 days.

Priority 1 — Do these before anything else

  • NOWChange ALL passwords: root, local admins, domain accounts, service accounts, Palo Alto (admin/Changeme123), VyOS router
  • NOWFirewall: set egress to DENY ALL by default, then whitelist only required outbound
  • NOWFirewall: allow only scored service ports inbound (80, 443, 25, 110, 53, 21, 69/udp, 123/udp)
  • NOWAudit /etc/passwd — remove or lock unknown accounts on every Linux box
  • NOWCheck ~/.ssh/authorized_keys on ALL Linux boxes including root — remove unknown keys
  • NOWCheck Windows AD — remove unknown users, audit Administrators group membership
  • NOWEnable Windows Firewall on all Windows boxes, remove all exceptions

Priority 2 — Complete within first 15 minutes

  • SOONBaseline running processes: ps aux > /tmp/baseline_ps.txt on each Linux box
  • SOONBaseline open ports: ss -tulpn / netstat -ano on every machine
  • SOONAudit crontabs: crontab -l for all users, check /etc/cron*
  • SOONStart watching auth logs: tail -f /var/log/auth.log and Windows Event Viewer (Security)
  • SOONVerify all scored services are UP and passing checks in the NISE Team Portal
  • SOONCheck /tmp/ and hidden directories for unknown files or binaries

Priority 3 — Complete within first 30 minutes

  • SOONReply to the Welcome inject in NISE (required — signals rule acceptance)
  • SOONFind SUID binaries and note them as baseline: find / -perm -4000 -type f 2>/dev/null
  • SOONCheck for immutable files (red team trick): find /etc/ -exec lsattr {} 2>/dev/null \; | grep '^....i'
  • SOONSet up log monitoring — at minimum tail -f on auth logs on each machine
  • SOONPrint and distribute password sheets for Day 1 (pre-generated, use words, not random chars)

03

Scored Services — What Gets Checked

The scoring engine checks services at random intervals. Services must be functional and serve their intended business purpose — not just open ports. SLA penalty accrues if services are down too long.

ServicePortHow It's ScoredNotes
HTTP80/TCPSpecific page requested, content compared to expected result byte-for-byteDon't change web content without knowing what scorer expects
HTTPS443/TCPSame as HTTP but over SSL — content must match expected fileSSL cert errors may or may not matter — verify with White Team
SMTP25/TCPEmail sent and received via a valid account — simulates employee in the fieldBoth send AND receive must work
POP3110/TCPConnections made with AD usernames, commands run, output examinedRequires AD users to exist and be functional
DNS53/UDP+TCPLookups performed — each successful response = pointsHigh frequency — keeping this up is easy points
FTP21/TCPAuth or anonymous access made, presence of specific files checkedEnsure expected files exist; anonymous may need to be enabled
TFTP69/UDPPull a small file with an integrity checkMake sure the expected file is present and unchanged
NTP123/UDPTime integrity checkEnsure NTP service is running and syncing correctly
Critical: You MUST NOT block the scoring engine's IP or restrict it via ACLs. You may block single confirmed-malicious IPs, but blocking entire subnets or placing a knife-edge ACL to allow only the scoring engine are both prohibited and will directly reduce your score.

Team Roles & Chain of Command

Scope creep kills teams. Know your role and stay in your lane unless explicitly asked to help elsewhere. The list below is in order of operational importance.

🎯 Team Captain

  • Coordinates, communicates — STAYS OFF THE KEYBOARD
  • Goes to all CEO inject meetings — never empty-handed
  • Carry a 1-page team status report: injects status, compromises found/cleaned, member tasks, future plans
  • Constantly asks team for feedback on tasks
  • Stops infighting, channels all feedback
  • Does NOT need to be the most technical person

📋 Executive Assistant / Runner

  • Handles all paperwork-based injects
  • Downloads anything the team needs
  • Steps in for Captain when absent
  • Supports all other roles as needed
  • Inherits physical security responsibilities
  • Gets food / supplies

🔥 Firewall Admin

  • First task: RAISE SHIELDS — egress deny all, whitelist scored services
  • Monitor ALL outbound connections constantly
  • Egress and ingress filtering
  • IPv6 OFF unless required by inject
  • Palo Alto config — change creds immediately
  • Pass incident reports to IR person

🐧 Linux Admin

  • Change all passwords immediately
  • Audit users, crontabs, SSH keys
  • iptables hardening, Fail2Ban
  • Watch /var/log/auth.log live
  • Remove unused users and services
  • Baseline and diff process lists

🪟 Windows Admin

  • Bulk AD password reset immediately
  • GPO: NTLMv2 only, disable LM hash, audit logging
  • Enable Windows Firewall, remove all exceptions
  • Remove unused users and services
  • Autoruns + Process Explorer monitoring
  • Rename default Administrator account

🌐 Web Admin

  • Find and change all web app passwords
  • Restrict admin interfaces to localhost/admin box
  • Watch access logs — ship to syslog if possible
  • PHP hardening: disable exec, passthru, shell_exec
  • Know what content the scoring engine expects

🔍 Incident Responder

  • Monitor logs on all systems
  • Fill out IR forms for every confirmed compromise — big points
  • Document everything: what happened, how, when, what changed
  • Check .bash_history, authorized_keys, lsof/netstat
  • Windows: Autoruns, qwinsta, Event Viewer Security
  • Eradicate access, confirm clean, document

Linux Hardening — Commands

Immediate — accounts and access

# Change root password passwd root # List all accounts with shell access cat /etc/passwd | grep -v nologin | grep -v false | grep -v sync # Audit all SSH authorized_keys (run as root) find / -name authorized_keys 2>/dev/null find / -name authorized_keys -exec cat {} \; 2>/dev/null # Check all crontabs crontab -l for user in $(cut -f1 -d: /etc/passwd); do crontab -u $user -l 2>/dev/null; done ls -la /etc/cron* /var/spool/cron/

Baseline running state

# Save process baseline (do this at start of EACH day) ps aux > /tmp/baseline_ps_$(date +%H%M).txt # Save open port baseline ss -tulpn > /tmp/baseline_ports.txt # Active connections netstat -antp

Firewall with iptables

# Save existing rules first iptables-save > /tmp/iptables_backup.rules # Basic lockdown — adjust ports to match your scored services iptables -P INPUT DROP iptables -P FORWARD DROP iptables -P OUTPUT DROP # Allow established connections iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT # Allow loopback iptables -A INPUT -i lo -j ACCEPT iptables -A OUTPUT -o lo -j ACCEPT # Allow scored services inbound (add/remove as needed) iptables -A INPUT -p tcp --dport 80 -j ACCEPT # HTTP iptables -A INPUT -p tcp --dport 443 -j ACCEPT # HTTPS iptables -A INPUT -p tcp --dport 25 -j ACCEPT # SMTP iptables -A INPUT -p tcp --dport 110 -j ACCEPT # POP3 iptables -A INPUT -p tcp --dport 53 -j ACCEPT # DNS TCP iptables -A INPUT -p udp --dport 53 -j ACCEPT # DNS UDP iptables -A INPUT -p tcp --dport 21 -j ACCEPT # FTP iptables -A INPUT -p udp --dport 69 -j ACCEPT # TFTP iptables -A INPUT -p udp --dport 123 -j ACCEPT # NTP # Allow SSH (management — consider restricting by source IP) iptables -A INPUT -p tcp --dport 22 -j ACCEPT # Allow outbound DNS, HTTP, NTP (minimum for updates) iptables -A OUTPUT -p udp --dport 53 -j ACCEPT iptables -A OUTPUT -p tcp --dport 80 -j ACCEPT iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT iptables -A OUTPUT -p udp --dport 123 -j ACCEPT # Save rules iptables-save > /etc/iptables/rules.v4

SSH hardening

# Edit /etc/ssh/sshd_config — key settings PermitRootLogin no PasswordAuthentication yes # keep yes unless you have keys distributed MaxAuthTries 3 AllowUsers youradminuser # whitelist specific users # Restart SSH systemctl restart sshd

Find red team persistence

# Check for immutable files (common red team trick) find /etc/ -exec lsattr {} 2>/dev/null \; | grep '^....i' # Remove immutable flag: chattr -i /path/to/file # Check /tmp for new binaries or hidden files ls -la /tmp/ find /tmp -executable -type f 2>/dev/null # Check for hidden directories in common locations ls -la ~/ /root/ /home/*/ # Find all SUID binaries and compare to baseline find / -perm -4000 -type f 2>/dev/null | sort > /tmp/suid_now.txt # diff /tmp/suid_baseline.txt /tmp/suid_now.txt # Find recently modified files in key directories find /etc /bin /sbin /usr -newer /tmp/baseline_ps.txt -type f 2>/dev/null # Check bash history cat ~/.bash_history cat /root/.bash_history

Palo Alto default credentials: admin / Changeme123 — Change these FIRST before anything else on the firewall. Both Linux FW and Windows FW.

Core firewall strategy (from red team)

What stops us mostHow to implement it
#1: Strong Egress Filtering — stops callbacks, alerts you to activity, forces you to think about what's talkingDefault DENY outbound. Whitelist only: DNS (53), HTTP/S for updates (80/443), NTP (123), scored service responses. Everything else drops.
#2: Good Ingress Filtering — only allow scored services, everything else DENYAllow only the 8 scored service ports. Deny all else. Double-check periodically — red team may have changed rules.
#3: Watching Logs — red team says most teams don't actually look at logsSomeone must have firewall logs open. Any unexpected outbound connection = immediate investigation.

Allowed inbound ports (scored services)

TCP 80 HTTP TCP 443 HTTPS TCP 25 SMTP TCP 110 POP3 UDP 53 DNS TCP 53 DNS (also allow TCP) TCP 21 FTP (also consider passive FTP ports if needed) UDP 69 TFTP UDP 123 NTP

Critical rules — read carefully

  • NEVER block entire IP subnets — rules violation and scoring penalty
  • NEVER place a knife-edge ACL at ingress to allow only the scoring engine — prohibited
  • You MAY block single confirmed-malicious IPs — report via Support Ticket
  • Traffic generators send suspicious traffic from random IPs — do NOT block those subnets
  • Scoring engine MUST always be able to reach your services — any block = direct point loss
  • IPv6: disable unless required by an inject
  • Monitor outbound constantly — unexpected egress = red team callback = you're compromised

Web / Application Hardening

HTTP and HTTPS are content-match scored — the engine compares returned page content to an expected file. Do NOT change web content without first knowing what the scoring engine expects. When in doubt, ask the White Team.

Immediate steps

  • Find all web application admin interfaces (/admin, /wp-admin, /phpmyadmin, /manager) and restrict to localhost or admin machine only
  • Change ALL web app passwords — red team found them before you did
  • Start watching access logs live: tail -f /var/log/apache2/access.log
  • Understand what the web app does in the context of the business scenario

PHP hardening (if applicable)

# Edit /etc/php/*/php.ini disable_functions = exec,passthru,shell_exec,system,popen,proc_open,curl_exec expose_php = Off display_errors = Off log_errors = On allow_url_include = Off

Apache / Nginx hardening

# Hide server version ServerTokens Prod ServerSignature Off # Disable directory listing Options -Indexes # Check for mod_security (install if not present — ask White Team for approval) apachectl -M | grep security

Injects & Business Tasks

Injects are worth 35–50% of your total score — equal to service uptime. Missing injects to chase technical problems is one of the most common ways teams lose. Assign a dedicated person to monitor NISE at all times.

Inject checklist

  • NOWReply to Welcome inject at 8:00 AM on Day 1 — required, signals rule acceptance
  • Complete pre-competition survey (every team member)
  • Assign runner / executive assistant to own all paperwork injects
  • Monitor NISE Team Portal constantly — all injects, scoring, and service status are there
  • Captain goes to CEO meetings — always carry a 1-page status report
  • Submit Incident Response reports for every confirmed compromise — this is a significant point source
  • Use Support Services for all requests: software downloads, service checks, system resets, password changes
  • Always identify by team number, NEVER by school name
  • Prepare a Disaster Recovery Plan for a critical service — this is a known inject type

Competition Rules — Must-Know

Instant disqualification: Any offensive activity (port scans, vulnerability scans, unauthorized connections) against other teams, White Team, or Red Team.

Technical rules

  • You MAY move a service to another platform — same public IP and DNS naming required
  • You MAY containerize a service (note NCCDC rule 4n if advancing)
  • You MUST maintain public services accessible from nominally all source IPs
  • You MAY block single confirmed-malicious IPs — not subnets
  • Do NOT change internal IP addressing unless directed by inject
  • Do NOT change system names unless directed — affects scoring
  • Workstations may NOT be repurposed
  • No peer-to-peer file sharing unless specifically authorized

Access / communication rules

  • No external communication — no calls, emails, chat, forum posts with non-team members
  • No private external FTP sites, staging areas, Google Drive, or personal cloud storage
  • All internet resources used must be freely available to all teams
  • GitHub repos will be copied to Nextcloud — accessible to all teams, use that
  • No USB drives, CD-ROMs unless authorized by Black/White Team in advance
  • Printed reference materials (books, checklists) ARE permitted

NETLAB access rules

  • DO NOT accidentally end your NETLAB reservation — this resets your entire environment
  • Change NETLAB password after first login and document it immediately
  • System restores reset all vulnerabilities AND negatively impact scores — avoid requesting them
  • Green Team and White Team will access your VMs for scoring — do not be alarmed, it won't hurt you

Red Team Intel — What Actually Stops Them

From the red team's own slides: "You don't need fancy tools. You need speed, discipline and focus." The basics done fast win every time.

What the red team says works against them (do these)

DefenseWhy it works
Strong egress filteringStops callbacks. Alerts you to their activity. Forces you to think about what's talking outbound.
Good ingress filteringOnly allow scored services — everything else DENY by default. Check periodically.
Changing passwords fastDo it fast, do it often. Automate it. Include local admin accounts AND computer accounts. Create new admins and switch.
Actually watching logs"We do not have the time to worry about them most of the time." Watching logs pays dividends.
Network traffic awarenessKnow what expected traffic looks like. Kill/block anything that isn't a scored service connection.

What the red team says does NOT stop them (don't waste time)

  • Patching everything — patch only what is actually exploitable, maximize impact per time spent
  • Complex SIEM setups — you don't have time to install and configure them, much less watch them
  • Threat hunting — you're blue team, focus on the basics
  • "Next gen" anything — simple tools done well beat fancy tools done poorly
  • Snort / Splunk / Nagios — no time to configure and watch them

What the red team actually uses to get in

  • Default credentials — this is their #1 method, not 0days or advanced tools
  • Default web app admin passwords
  • Pre-placed backdoors and SSH keys from before flag drop
  • Metasploit psexec, MS08_067, MS09_050 (classic Windows exploits)
  • Persistence via scheduled tasks, registry run keys, services
0day reminder: No one is going to drop 0day at CCDC. Patch what is exploitable and known, not everything. Prioritize speed over completeness.
